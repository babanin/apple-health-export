# AGENTS.md

## Project Overview

Apple Health Export syncs 52 HealthKit metrics from iPhone to Victoria Metrics (time-series DB) via a Python gRPC gateway, visualized in Grafana.

## Tech Stack

- **iOS App**: Swift 6, SwiftUI, iOS 18+, HealthKit, grpc-swift (NIO transport + protobuf)
- **Gateway**: Python 3.12, grpcio, protobuf, requests
- **Storage**: Victoria Metrics (NDJSON import API), SQLite (checkpoints), UserDefaults (client checkpoints)
- **Visualization**: Grafana with auto-provisioned Apple Health dashboards
- **Containerization**: Docker Compose (3 services: VM, Grafana, Gateway)
- **Testing**: pytest (58 tests across 7 files), XcodeGen for project generation

## Key Conventions

### Swift (iOS App)
- All files in `ios-app/HealthExporter/` — auto-included by XcodeGen via `project.yml` sources path
- Target: iOS 18.0, Swift 6.0
- Minimum deployment: iOS 18.0 (required for gRPC-Swift NIO transport)
- `@MainActor` on all UI-related observable classes
- `AppConfig.shared` singleton for runtime config (UserDefaults-backed, editable in UI)
- `AppLogger.shared` singleton for in-app logging (thread-safe, publishes to `@Published var entries`)
- `CheckpointManager.shared` singleton for client-side checkpoint persistence
- `DemoDataManager.shared` singleton for fake health data
- Health metric porting pattern: add entry to `HKMetricMapping.all` array in `Models/HKMetricMapping.swift`
- gRPC stubs in `Generated/` are auto-generated — do not edit manually, regenerate with `make proto-swift`
- XcodeGen project auto-includes all `.swift` files under `HealthExporter/` — just add files, no project file editing needed
- After regenerating the Xcode project with `xcodegen generate`, always build to verify compilation:
  ```bash
  cd ios-app && xcodegen generate && xcodebuild -scheme HealthExporter -sdk iphonesimulator -destination 'platform=iOS Simulator,name=iPhone 17,OS=latest' build
  ```
- Or build for a connected physical device:
  ```bash
  xcodebuild -scheme HealthExporter -sdk iphoneos -destination 'platform=iOS,name=<device-name>' build
  ```

- `GrpcClient` must call `grpc.runConnections()` in a detached task after creating the `GRPCClient` — without it, RPCs silently hang until timeout. The `connect()` method stores the `GRPCClient` in `self.grpcClient` and starts `Task.detached { try await grpc.runConnections() }`. The `disconnect()` method calls `grpcClient?.beginGracefulShutdown()` and `connectionTask?.cancel()` before niling out references.

### Python (Gateway)
- All gateway code in `gateway/`
- gRPC stubs (`health_export_pb2.py`, `health_export_pb2_grpc.py`) are auto-generated — edit `proto/health_export.proto` then `make proto`
- VMWriter uses NDJSON format with periodic (1s) and threshold (5000 samples) flush
- CheckpointStore uses SQLite with UPSERT + MAX for monotonic timestamps
- Server runs 10 worker threads, 100MB max message size

### Protobuf
- Package: `health_export`
- Go package option: `github.com/apple-health-export/proto/health_export`
- Proto3 syntax

### Grafana
- Curated dashboards are generated by `scripts/generate_dashboards.py`; do not hand-edit generated dashboard JSON unless making an emergency one-off fix.
- Run `make dashboards` after dashboard generator changes. It writes JSON under `dashboards/` and mirrors it into `grafana/provisioning/dashboards/`.
- Grafana datasource provisioning must keep the Victoria Metrics datasource UID as `victoriametrics`; dashboards reference this concrete UID. Do not use `${DS_VICTORIAMETRICS}` in provisioned dashboards.
- `grafana/provisioning/datasources/datasource.yml` deletes and recreates `Victoria Metrics` during provisioning so existing Grafana volumes get the stable UID.
- Sparse slow-moving health metrics should be visualized with a forward-filled PromQL query, e.g. `last_over_time(apple_health_vo2_max_ml_kg_min{source=~"$source"}[$__range])`, with `spanNulls` enabled. This prevents charts such as VO2 Max, resting HR, blood pressure, and body measurements from showing broken gaps between HealthKit samples.
- Dense noisy health metrics should keep a trend view instead of forward-fill. HRV SDNN uses rolling averages such as `avg_over_time(apple_health_heart_rate_variability_ms{source=~"$source"}[7d])` and `[30d]` so long-range dashboards show tendency instead of only raw noise.

## Architecture: Data Flow

```
iPhone HealthKit
  → HealthKitManager.fetchAllMetrics() [52 types, checkpoint-aware start dates]
  → SyncManager.startSync() [batches of 500]
  → GrpcClient.syncMetrics() [HTTP/2 plaintext, configurable host:port]
  → HealthExportServicer.SyncMetrics() [Python gRPC server, port 50051]
  → VMWriter.buffer() + periodic flush [NDJSON to /api/v1/import]
  → Victoria Metrics [10yr retention, port 8428]
  → Grafana [port 3000, PromQL queries]
```

## Checkpoint Protocol

1. Client sends `SyncRequest.checkpoint` (per-metric millisecond timestamps from UserDefaults)
2. Server UPSERTs into SQLite `checkpoints(device_id, metric_type, last_timestamp_ms)`, taking `MAX(existing, new)`
3. Server returns `SyncResponse.updated_checkpoint` with merged values
4. Client updates local UserDefaults from response
5. Next sync: `HealthKitManager` uses `CheckpointManager.getStartTime()` → only fetches new data

## iOS App: In-App Logging

`AppLogger.swift` — thread-safe `@MainActor` singleton:
- Levels: `.debug` (gray), `.info` (blue), `.warning` (orange), `.error` (red)
- Auto-includes `[Filename:line]` prefix
- 500 entry cap, auto-scroll in `LogPanelView`
- Clear button to wipe log buffer
- Access: `AppLogger.shared.info("message")` from any thread

## Metric Naming Convention

All metric names use prefix `apple_health_` with underscores and unit suffix:
- `apple_health_heart_rate_bpm`
- `apple_health_steps_total`
- `apple_health_blood_glucose_mg_dl`
- `apple_health_sleep_stage` (category type, value is stage index, `labels.stage` holds name)

## Common Gotchas

- iOS app requires iOS 18.0+ (gRPC-Swift NIO transport requirement)
- gRPC uses plaintext HTTP/2 — no TLS in current config
- HealthKit requires paid Apple Developer account for real data; demo mode works without
- `Generated/` directory contains proto-generated Swift files — do not edit, regenerate with `make proto-swift`
- XcodeGen project auto-includes all `.swift` files under `HealthExporter/` — just add files, no project file editing needed
- `AppLogger.shared` must be accessed from `@MainActor` when reading `entries` for UI; writing from any thread is safe (dispatches to MainActor internally)
- `AppConfig.shared` is `@MainActor` — read/write from main thread, or use `Task { @MainActor in ... }`
- Swift 6 strict concurrency: avoid capturing `var` in `@Sendable` closures. Use a class wrapper (`@unchecked Sendable`) for mutable shared state like guards/locks
- `NSLocalNetworkUsageDescription` is required in Info.plist — without it, iOS silently blocks local network connections
- After any Swift code change, always run `xcodegen generate && xcodebuild ... build` to verify compilation before asking user to test
- Grafana only reloads provisioned datasources/dashboards on startup; after dashboard or datasource provisioning changes, run `docker compose restart grafana`
- If Grafana reports `Datasource provisioning error: data source not found` after changing datasource UID, preserve the `deleteDatasources` block and stable `uid: victoriametrics` in `grafana/provisioning/datasources/datasource.yml`
